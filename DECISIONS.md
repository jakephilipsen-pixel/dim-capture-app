| 2026-06-03 | **Production is single-origin**: the frontend is built with a *relative* API base (`VITE_API_URL=""` build-arg ‚Üí `api.ts`'s `?? 'http://localhost:3005'` yields `/api`), and the frontend's nginx proxies `/api/*` ‚Üí `backend:3005` over the compose network. Caddy fronts `dims.gocold.local` ‚Üí frontend:80 only; the backend is never exposed to the browser. **This overrides the spec's `VITE_API_URL=http://backend:3005`** (spec DOCKER COMPOSE block), which is broken two ways: it sets a *build-time* Vite var as a *runtime* `environment:` (Vite never sees it), and even baked in, the browser cannot resolve the Docker-internal hostname `backend`. Confirmed with Jake before building. | docker-deploy build |
| 2026-06-03 | nginx's `/api/` proxy uses a **variable upstream** (`set $backend_upstream backend:3005; proxy_pass http://$backend_upstream;`) + `resolver 127.0.0.11` | A literal `proxy_pass http://backend:3005` makes nginx resolve the upstream at *startup* and refuse to boot when no `backend` exists ‚Äî which would break the frontend-only `frontend-scaffold` smoke. A variable defers DNS to request time, so nginx always boots; `/api` only errors if actually called (the frontend-only smoke never calls it) | docker-deploy build |
| 2026-06-03 | Production deploy URL is **`http://dims.gocold.local`** (LAN-only, host-Caddy ‚Üí published frontend port 5175), superseding the scaffold default `dim-capture-app.rolodex-ai.com`. Caddy fragment lives at `caddy/dims.gocold.local.caddy`, installed to `/etc/caddy/Caddyfile.d/` on the NUC; local DNS must resolve `dims.gocold.local` ‚Üí NUC IP (one-time network setup, documented in NUC_DEPLOY.md, not automated) | brief + spec both specify `dims.gocold.local`; no Cloudflare tunnel for this project | docker-deploy build |
| 2026-06-04 | `POST /api/sync/cc` serialised via `pg_try_advisory_xact_lock` in an interactive transaction (auto-release, single pooled connection); the loser returns `{synced:0,failed:0,pending:<live count>}` without PATCHing | Prevents concurrent runs from both reading the same unsynced set and double-PATCHing the REAL CartonCloud (over-reporting `synced` and burning CC's 60/min token bucket so legit syncs get rejected). A session-level `pg_advisory_lock`+`pg_advisory_unlock` pair would leak the lock forever, because Prisma pools connections and the unlock can land on a different connection than the lock | compile-stress C1 fix |
| 2026-06-08 | The CC-write routes ‚Äî `POST /api/sync/cc` and `POST /api/admin/seed` ‚Äî require a **shared-secret header** (`X-Sync-Key` matching env `SYNC_SECRET`); missing/empty/wrong ‚Üí **401** before any handler logic runs. If `SYNC_SECRET` is unset the routes refuse (fail-closed, 503). No other routes are gated. Read routes (`GET /api/*`) and dim capture (`POST/PUT /api/dims`) stay open on the LAN | Satisfies the parent gocold-wms-flow non-negotiable "never push to CC automatically ‚Ä¶ human approval gate" with the least surface: stateless, works behind the single-origin Caddy/nginx proxy, trivial on the NUC. Chosen by Jake 2026-06-08 over nginx basic-auth, an explicit confirm-step, and secret+confirm | pre-build decision for module 12 `cc-write-authz` |
| 2026-06-08 | Dim upper bounds: **100,000 mm** for L/W/H; **1,000 kg** for weight | 100 m covers any realistic carton in a cold-store 3PL environment; 1,000 kg is the Australian B-double pallet legal maximum. Both are large enough to never reject a real measurement, small enough to catch obvious data-entry errors and JSON injection (e.g. `1e308`) | backend-error-hardening build (M5) |
| 2026-06-08 | Zod v4 `Infinity` override via the `error` **callback** on `z.number(...)`, not `.finite()` | In zod v4's classic API `.finite()` is a no-op (`return this`). Non-finite inputs hit the type-level parse and produce the hardcoded opaque message "Invalid input: expected number, received number". The `error` callback is the documented v4 override point; we detect `!Number.isFinite(input) && typeof input === "number"` and return the field-specific "must be a finite number" message | backend-error-hardening build (M6) |
| 2026-06-08 | S3a: `CcApiError` ‚Üí **502**, `CcRateLimitError` ‚Üí **503**, caught and re-wrapped as `AppError` in **`skuService.getSkuByBarcode`** (not in `errorHandler`) | Keeps CC error-type knowledge in the service layer alongside the CC client import; `errorHandler` stays CC-agnostic. Follows the established pattern (services throw `AppError`; `errorHandler` only needs to know about `AppError` vs unknown). `CcApiError` = upstream problem = 502 Bad Gateway; `CcRateLimitError` = upstream unavailable = 503 Service Unavailable | backend-error-hardening build (S3a) |

| 2026-06-08 | S5: drop the backend host port entirely (not rebind to `127.0.0.1:3005`) | The backend has no legitimate host-side caller: the docker-deploy design documents it is not exposed to the browser; host health checks reach it via the nginx proxy (`localhost:5175/api/health`). Dropping the mapping is cleaner and strictly more secure than rebinding. `.health-endpoints` updated to the proxy path. | deploy-hardening build (S5) |
| 2026-06-08 | S7 CSP: `default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; connect-src 'self'; worker-src 'self'; manifest-src 'self'; media-src 'self'; frame-ancestors 'none';` ‚Äî no `'unsafe-inline'`, no `blob:` | Vite builds hashed bundles with no inline scripts or styles. ZXing assigns the camera stream via `srcObject` (not `createObjectURL`), so `blob:` is not required in `media-src`. `data:` in `img-src` allows SVG data URIs that React component libraries may emit. Verified by source inspection of `@zxing/browser` and the Vite/Tailwind build output. | deploy-hardening build (S7) |

## How to add a decision
